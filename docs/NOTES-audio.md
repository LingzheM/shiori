# 実施ノート — 章节音频（朗読）v1

> 2026-07-18 · 对应盲区扫描与反向采访的落地记录

## 关键决策与理由

**D1 · 音频引擎是模块单例，不是 React 组件状态。**
Reader 按章节 `key` 重挂载，音频元素若在组件树内会随之销毁。全局唯一 `<audio>` 元素还有第二个作用：iOS 上同一元素被用户手势解锁一次后，后续程序化 `play()`（续播下一章）也被放行。

**D2 · "无拖拽" ≠ "无位置记忆"。**
每 5 秒 + 暂停时 + 页面隐藏时（`visibilitychange`/`pagehide`）持久化 `currentTime`，键为 `audio-pos:{book}:{chapter}`。恢复规则：位置 > 5s 且距结尾 > 5s 才续播，否则从头。播完清零（下次是重听）。

**D3 · 变速进了 v1**（采访时用户委托决策）。
单击胶囊循环 1.0 → 1.25 → 1.5 → 0.8。全局持久化（`audio-rate`），跨章节生效。

**D4 · 进度线是信息不是控件。**
发丝线用 `transform: scaleX()` 更新（不动 width，性能纪律）。v2 做拖拽时同一元素原地升级，布局不变。

**D5 · 切章即停，位置已存。**
音频归属章节。翻章时旧章音频暂停并存位；回来按播放从原位继续。

**D6 · 播完提示自订阅。**
`timeupdate` 每 500ms 广播一次；若 Reader 本体订阅会导致整个阅读视图每秒重渲染两次。提示卡片（EndedPrompt）与朗读条各自订阅，Reader 零感知。

**D7 · Chrome 隐藏时音频继续。**
朗读条随工具栏显隐（沉浸阅读无 UI 噪音），声音不断；锁屏由 Media Session 接管（标题=章节名、作者=书名、播放/暂停可用）。

## ⚠️ 给你的生成脚本的一个请求（现在做，v2 免费）

生成每章音频时，**顺手输出句级时间戳 sidecar**，即使 v1 完全不用：

```
public/books/tokyo-vice/chapter1.ja.timing.json
[
  { "i": 0, "start": 0.00,  "end": 4.32,  "text": "最初の文……" },
  { "i": 1, "start": 4.32,  "end": 9.87,  "text": "次の文……" }
]
```

来源方式取决于你的 TTS：逐句合成再拼接的话，时长即时间戳；整章一次合成的话，检查 API 是否返回 word/sentence marks（Azure、ElevenLabs 等都有）。**没有这份数据，v2 句级播放要靠强制对齐，成本高一个量级；有了它，v2 只是查表。**

音频格式建议：**MP3 单声道 64–96kbps**（人声足够、体积减半、Safari 最稳）。manifest 章节对象加一个可选字段即可：

```json
{ "id": "01", "title": "第一章", "file": "books/tokyo-vice/chapter1.ja.txt",
  "audio": "books/tokyo-vice/chapter1.ja.mp3" }
```

无 `audio` 字段的章节，朗读条整个不出现。

## 真机验证清单（请在你的手机上勾）

- [ ] 点播放前不产生大流量（仅 metadata；蜂窝网络下确认）
- [ ] 播放中锁屏 → 声音继续；锁屏界面显示章节名，暂停键有效
- [ ] AirPods/耳机线控的播放暂停有效
- [ ] 听到中途杀掉页面重开 → 时间显示为上次位置，播放从该处继续
- [ ] 播完 → 底部浮出续播卡片；点「次の章を聴く」→ 切章且自动开播
- [ ] 十秒不理会卡片 → 自动淡去
- [ ] 变速切到 0.8x 后翻章 → 速度保持
- [ ] 无音频的章节 → 底栏无朗读条、无残留
